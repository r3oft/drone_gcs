# 物流无人机地面站软件 —— 初期开发计划

> 编制时间：2026-03-23 ｜ 修订时间：2026-04-09
> 版本：v0.3（单摄像头 + 取货区方案修订）
> 定位：在**硬件/飞控方案尚未冻结**的阶段，先行推进"封装层级高、硬件解耦度强"的通用工具链与中间件，为后续联调做准备。
>

## 一、 技术全景与开发分工

### 1.1 系统软件拓扑回顾

整个地面站程序由五大模块级联组成（详见 `算法控制逻辑.md`）：

| 模块编号 | 名称 | 核心职责 | 对硬件的依赖度 |
|:---:|:---|:---|:---:|
| M1 | `ZeroLatencyStreamer` | 低延迟视频流异步采集（单摄像头，外侧安装） | **高** — 需要 ESP32-CAM 的真实 MJPEG 流地址和帧率 |
| M2 | `TargetPoseEstimator` | YOLO-OBB 推理 + 位姿特征提取（**双类别**：pickup_zone + delivery_zone） | **低** — 仅需 GPU + 权重文件 + 静态图像即可离线验证 |
| M3 | `VisualServoController` | PD 视觉伺服闭环控制律 | **低** — 纯数学运算，可完全离线开发和仿真测试 |
| M4 | `MavlinkCommander` | MAVLink 协议封装 & 飞控通信 | **高** — 需要 Pixhawk 在线 + UDP 链路 + 联调 |
| M5 | 全局状态机 / 容错调度 | FSM 跃迁 / 看门狗 / Failsafe | **高** — 依赖 M1-M4 全部就绪 + 实机环境 |

> **结论**：M2（视觉推理）和 M3（伺服控制）对硬件几乎零依赖，是初期开发的首选切入点。除此之外，有大量"围绕这些模块的辅助工具和通用基础设施"可以先行建设。

### 1.2 与硬件/飞控组的协作边界

| 你（软件/算法组）可以先做的 | 需要等硬件/飞控组确认后再做的 |
|:---|:---|
| YOLO-OBB 模型训练 & 推理管线 | M1：ESP32-CAM 的流地址格式、分辨率、帧率、编码方式 |
| 视觉伺服 PD 控制器编写 & 仿真 | M4：Pixhawk 固件版本、MAVLink 版本、UDP 端口号、AUX 通道号映射 |
| 配置管理系统 | M4：`SERVOx_FUNCTION` 等飞控参数的具体配置值 |
| 日志 & 性能监控框架 | M5：飞控心跳包的返回格式、自检具体校验项 |
| 仿真数据生成器 & Mock 接口 | 通信协议与透传格式最终版确认 |
| 大部分单元测试 | 实机联调与参数整定 |

### 3.1 配置管理器（`utils/config_manager.py`）

**为什么先做？** 所有模块都需要从外部读取参数（PD 增益、阈值、网络地址……）。如果不提前统一管理，后续每个模块各自硬编码，联调时改参数会非常痛苦。

**核心功能：**
1. 从 YAML 文件加载全局参数树
2. 支持嵌套 key 的点号访问（如 `config.servo.pickup_align.kp.x`）
3. 支持命令行参数覆盖（方便调试时快速改参）
4. （可选进阶）文件监听热重载

**接口设计：**
```python
class ConfigManager:
    def __init__(self, config_path: str = "config/default.yaml"):
        """加载 YAML 配置文件，构建参数树。"""

    def get(self, key: str, default=None) -> Any:
        """
        点号分隔的嵌套读取。
        示例: config.get("servo.pickup_align.kp.x", 0.003)
        """

    def override_from_args(self, cli_args: dict) -> None:
        """用命令行参数字典覆盖已加载的配置值。"""

    def reload(self) -> None:
        """重新从文件加载配置（热重载）。"""

    def to_dict(self) -> dict:
        """返回当前完整配置的扁平字典（用于日志记录）。"""
```

**默认配置模板（`config/default.yaml`）：**

### 3.2 日志与黑匣子框架（`utils/logger.py`）

**为什么先做？** 飞行调试不像桌面程序可以随时断点。必须有完整的日志链路，否则出了问题完全没有追溯手段。

**核心功能：**
1. 基于 Python `logging` 的分级日志（DEBUG / INFO / WARNING / ERROR）
2. 同时输出到**控制台**（彩色）和**文件**
3. **飞行黑匣子**模式：以 CSV / JSON Lines 格式按时间戳逐帧记录关键遥测数据（状态、速度指令、视觉位姿、控制量），供事后回放分析
4. 日志文件以启动时间戳命名，自动归档到 `logs/` 目录

**接口设计：**
```python
def setup_logger(name: str, level: str = "INFO", log_dir: str = "logs/") -> logging.Logger:
    """创建并返回一个配置好的 Logger 实例。"""

class FlightRecorder:
    """飞行数据黑匣子。"""

    def __init__(self, log_dir: str = "logs/", fields: list[str] = None):
        """初始化记录器，指定 CSV 表头字段。"""

    def record(self, data: dict) -> None:
        """写入一条帧数据记录。自动附加 ISO 时间戳。"""

    def close(self) -> None:
        """刷新缓冲区并关闭文件句柄。"""
```

**黑匣子记录的字段示例：**
```
timestamp, state, target_u, target_v, target_theta, target_conf, cmd_vx, cmd_vy, cmd_vyaw, dt
```

**验收标准：**
- 控制台输出带颜色分级
- 日志文件以时间戳命名
- FlightRecorder 能正确写入 CSV 并可被 pandas 正常读取

### 3.3 几何运算工具（`utils/geometry.py`）

**为什么先做？** OBB 角度归一化是 M2（推理模块）和 M3（伺服控制）共用的核心数学操作。提前封装好、写好测试，后面用起来不会出错。

**核心功能：**
1. OBB 角度归一化（$C_2$ 对称折叠）：将任意角度映射到 $[-\frac{\pi}{4}, \frac{\pi}{4}]$ 或 $[-\frac{\pi}{2}, \frac{\pi}{2}]$
2. 像素误差到机体系误差的坐标映射
3. 死区函数（Deadband）
4. 饱和限幅函数（Saturation Clamp）

**接口设计：**
```python
def normalize_obb_angle(theta: float, symmetry_order: int = 2) -> float:
    """
    将 OBB 输出角度折叠到最小等效区间。
    symmetry_order=2 → C2对称 → 折叠到 [-π/4, π/4]
    symmetry_order=1 → 不折叠 → [-π/2, π/2]
    """

def pixel_to_body_error(u: float, v: float, center_u: float, center_v: float) -> tuple[float, float]:
    """
    将像素坐标误差映射为机体系 (前向, 侧向) 误差。
    基于下视 BEV 正交投影假设：
      E_x = center_v - v  (纵向像素差 → 前向)
      E_y = u - center_u  (横向像素差 → 侧向)
    """

def apply_deadband(value: float, threshold: float) -> float:
    """若 |value| < threshold，返回 0；否则返回原值。"""

def clamp(value: float, min_val: float, max_val: float) -> float:
    """硬饱和限幅。"""
```

**验收标准：**
- 单元测试覆盖各种边界角度（0, ±π/4, ±π/2, ±π）
- 确保 `normalize_obb_angle(π/2 + 0.01, symmetry_order=2)` 折叠到正确范围
- `apply_deadband(10, 15) == 0`，`apply_deadband(20, 15) == 20`

### 3.5 性能监控器（`utils/perf_monitor.py`）

### 3.7 YOLO-OBB 推理管线（`core/perception.py`）—— M2

**为什么先做？** 虽然模型训练需要采集实际场景数据，但推理管线的代码框架可以先写好，用网上的预训练 OBB 权重或合成图片来测试流程。

**双类别检测说明（v0.3）：**

本项目使用 **一个 YOLO-OBB 模型同时检测两个类别**，由 FSM 状态决定当前检测哪一类：

| 类别 ID | 类别名 | 标记物外观 | 使用阶段 |
|:---:|:---|:---|:---|
| 0 | `pickup_zone`（取货区） | 蓝色胶带方形，~30×30cm | TASK_REC 阶段 |
| 1 | `delivery_zone`（投递区） | 红色胶带方形，~30×30cm | TASK_REL 阶段 |

> **切换时机**：取货区对准→人工装填→红外确认装填完成→定距转移→抵达投递区后切换到 `target_cls_id=1`。切换逻辑由 M5（FSM）驱动，M2 本身无需感知阶段切换。

**接口设计：**（与 `算法控制逻辑.md` 保持一致）
```python
class TargetPoseEstimator:
    def __init__(self, weights_path: str, conf_threshold: float = 0.6,
                 device: str = "cuda:0"):
        """加载 YOLO-OBB 模型，执行预热推理。"""

    def process_frame(self, frame: np.ndarray,
                      target_cls_id: int) -> dict | None:
        """
        输入: RGB 图像帧, 目标类别 ID（0=pickup_zone, 1=delivery_zone）
        输出: {"u", "v", "theta", "conf", "w", "h"} 或 None
        theta 已经过 C2 对称归一化。
        """
```

**初期验证方式：**
- 用 `ultralytics` 库内置的 `yolov8n-obb.pt` 预训练权重跑 DOTA 数据集的示例图
- 分别用 `target_cls_id=0` 和 `target_cls_id=1` 测试滤波逻辑
- 验证推理管线链路通畅、角度归一化正确
- 后期替换为自训练双类别权重即可

### 3.8 Mock 模拟层（`utils/mock.py`）

**为什么先做？** 在硬件不可用时，为 M1（视频流）和 M4（MAVLink）提供模拟替身，使主程序可以脱离硬件进行全链路桌面联调。

**核心功能：**
1. `MockStreamer`：从本地视频文件或图片序列模拟相机流
2. `MockMavlinkCommander`：打印/记录所有发送的指令而不实际通信
3. `MockMCU`：模拟红外传感器返回值（模拟装填完成/释放成功信号）

**接口设计：**
```python
class MockStreamer:
    """从本地视频/图片序列模拟摄像头流，接口与 ZeroLatencyStreamer 一致。"""
    def __init__(self, source: str):
        """source 可以是本地视频路径或图片目录。"""
    def get_latest_frame(self) -> np.ndarray | None: ...

class MockMavlinkCommander:
    """记录所有指令但不实际发送，接口与 MavlinkCommander 一致。"""
    def __init__(self):
        self.command_log: list[dict] = []
    def send_body_velocity(self, vx, vy, vz, yaw_rate) -> None: ...
    def set_gripper_state(self, is_open: bool) -> None: ...
    def get_command_log(self) -> list[dict]: ...
```

> **Mock 联调覆盖性要求**：Mock 测试必须覆盖完整的"对准取货区 → 等待装填 → 红外确认 → 定距转移 → 对准投递区 → 投递"流程。需要准备两套不同的图片序列（分别包含 pickup_zone 和 delivery_zone 标注）。

## 四、 单摄像头 + 取货区方案设计（v0.3）

### 4.1 方案概述

由于自主抓取货箱的难度过大，系统降级为以下方案：

1. **取货区替代货箱**：场地内用蓝色胶带粘出 ~30×30cm 方形"取货区"，无人机视觉对准取货区后降落
2. **人工装填**：由人手动将货箱装在无人机叉刀上
3. **红外确认装填完成**：红外模块检测到货物装填完成后，向地面站发送信号
4. **定距转移**：无人机起飞后按预设路径飞往投递区（此阶段视觉/PD 控制不工作）
5. **视觉对准投递区**：抵达投递区上空后，用同一个摄像头检测红色胶带投递区，PD 控制对准后释放货物

### 4.2 单摄像头优势

改为取货区方案后，**不再需要双摄像头**：
- 原方案需要双摄像头的原因是"抓取到货箱后，货箱遮挡中心摄像头视野"
- 新方案中货箱由人工装填到叉刀上，摄像头安装在机架外侧方，视野不受货箱遮挡
- 因此可以用**同一个外侧摄像头**在两个阶段分别检测取货区和投递区，大幅降低系统复杂度

### 4.3 FSM 状态 — 目标类别映射表

| FSM 状态 | 视觉/PD 工作？ | target_cls_id | 说明 |
|:---|:---:|:---:|:---|
| IDLE | 否 | - | 静置 |
| RESET | 否 | - | 舵机复位/自检 |
| INBOUND | 否 | - | 起飞/飞往取货区 |
| **TASK_REC_ALIGN** | **是** | **0 (pickup_zone)** | **视觉对准取货区** |
| TASK_REC_DESCEND | 是 | 0 (pickup_zone) | 盲降到取货区 |
| **TASK_REC_WAIT_LOAD** | **否** | - | **悬停等待人工装填，红外确认装填完成 ★ 新增** |
| TRANS_DELIVERY | 否 | - | 定距转移飞往投递区（视觉/PD 完全不工作） |
| **TASK_REL_ALIGN** | **是** | **1 (delivery_zone)** | **视觉对准投递区** |
| TASK_REL_DESCEND | 是 | 1 (delivery_zone) | 盲降到投递区 |
| TASK_REL_RELEASE | 否 | - | 执行释放/红外校验 |
| TRANS_CARGO | 否 | - | 飞往下一个取货区 |
| OUTBOUND | 否 | - | 返航/降落 |

> **关键流程**：`TASK_REC_DESCEND` → 降落后进入 `TASK_REC_WAIT_LOAD` → 人工装填 → 红外确认 → 起飞 → `TRANS_DELIVERY`（定距转移，视觉/PD 不工作）→ 抵达投递区后 → `TASK_REL_ALIGN`（**此时 target_cls_id 切换为 1**）

### 4.4 摄像头安装偏移补偿（Phase 3 联调时处理）

外侧摄像头的安装位置与机体中心可能不完全重合，会引入系统性的定位偏差。配置文件中已预留 `camera.offset` 字段，在实机联调时填入实测偏移量，并在 `pixel_to_body_error()` 映射时加入补偿。

## 七、 技术注意事项

### 7.1 关键设计约束

1. **无积分项（No-I）原则**：PD 控制器严禁引入积分环节，否则在通信延迟下会导致积分饱和和振荡
2. **角度归一化必须在推理之后、控制之前完成**：这是 M2 到 M3 数据流的关键衔接点
3. **机体系坐标**：所有速度指令必须基于 `MAV_FRAME_BODY_NED`，与地球磁场无关
4. **线程安全**：M1 涉及多线程共享帧缓冲，必须使用互斥锁防止图像撕裂
5. **阶段驱动目标切换**：`target_cls_id` 的切换由 M5（FSM）统一管理，M1/M2/M3 本身保持无状态

### 7.2 依赖库清单

## 八、 总结

初期开发的核心策略是：**"自底向上造轮子，由内而外搭框架"**。

1. **先做工具层**（配置管理、日志、性能监控、几何运算、可视化）—— 这些是所有后续开发的地基
2. **再做纯算法层**（视觉伺服控制器、YOLO 推理管线）—— 完全不依赖硬件，可离线完成开发和验证
3. **同步搭建 Mock 框架** —— 在没有硬件的情况下也能跑通完整 Pipeline，验证状态机逻辑
4. **最后对接实机** —— 等硬件方案冻结后，用真实模块替换 Mock，只需改配置和实现子类

这样的开发策略确保了：即使硬件组和飞控组的进度滞后，软件组也能持续推进有意义的工作，且后续联调时只需要做"拼接"而非"重写"。
